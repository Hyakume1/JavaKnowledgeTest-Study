Printing to the Screen

You already know System.out.println() — it prints something and then moves to the next line. Its sibling, System.out.print(), prints without moving to a new line.

System.out is a special object Java gives you automatically — you never need to import or create it. It's always ready to use.

Basic output
System.out.println("Hello, World!");   // prints and moves to next line
System.out.print("Enter a number: ");  // prints WITHOUT moving to next line
System.out.println(42);                // works with numbers too
System.out.println("Sum: " + (3 + 4)); // "Sum: 7"
Output 
Hello, World!
Enter a number: 42
Sum: 7

Both print() and println() are smart — they automatically convert any data type (int, double, boolean, etc.) into readable text for you. You don't have to do any converting yourself.

Buffering — Why Output Doesn't Always Appear Immediately

Here's something surprising: when your code calls System.out.print(), the characters are not immediately written to the screen! Instead, they go into a buffer — think of a buffer like a waiting room. Characters sit in that waiting room until something triggers them to actually appear on screen.

What triggers a flush (emptying the buffer)?

  • println() — always flushes because System.out has autoFlush turned on
  • print() with a \n in the string — also flushes (the autoFlush detects the newline)
  • flush() — calling System.out.flush() explicitly forces everything out
  • print() without any newline — does NOT flush; text just waits
Which lines appear on screen immediately?
// Poem by Henry David Thoreau
System.out.print("My life has been the poem ");  // A — BUFFERED (no newline)
System.out.println("I would have writ");          // B — FLUSHED immediately (println)
System.out.print("But I could not ");             // C — BUFFERED (no newline)
System.out.flush();                               // D — forces A+B+C out NOW
System.out.print("both live and utter it.\n");    // E — FLUSHED (contains \n)

So: Line A waits. Line B flushes (and carries Line A with it). Line C waits again. flush() forces Line C out. Line E flushes itself because the \n triggers autoFlush.

What Is System.out, Really?

System.out is an object of the class PrintStream. You never need to write import java.io.PrintStream; — it's part of the java.lang package, which Java loads automatically for every single program. Same goes for things like String and Math.

Reading Input from the User

To read what the user types, Java uses something called a Scanner. Think of Scanner like a helper that unpacks a stream of bytes and hands you back usable types — strings, ints, doubles — through methods like next(), nextInt(), and nextDouble().

You pass any InputStream to Scanner's constructor. System.in is just one example of an InputStream — it's the keyboard. You could also pass a FileInputStream to read from a file.

Scanner wraps an InputStream
import java.util.Scanner;

Scanner scnr = new Scanner(System.in);        // wrap keyboard input
// Scanner fileScnr = new Scanner(someFileInputStream);  // wrap a file too

What Is System.in, Really?

System.in is a predefined object of the class InputStream — it's connected to your keyboard. When you type at the keyboard and press Enter, the operating system collects your keystrokes into a buffer. System.in then reads from that OS buffer one byte at a time.

Just like System.out, you never need to import InputStream — it's also in java.lang, which is always auto-loaded.

throws IOException

When your program reads from a file (not the keyboard), things can go wrong that are out of your control — the file might not exist, or the disk might fail. Java requires you to acknowledge this possibility by adding throws IOException to your main() method.

Important: The clause is throws IOException — NOT throws InputException. InputException does not exist in Java!

Adding throws IOException for file reading
public static void main(String[] args) throws IOException {
    FileInputStream fs = new FileInputStream("myfile.txt");
    Scanner scnr = new Scanner(fs);
    // ...
}

When reading from the keyboard using Scanner(System.in), you do NOT need throws IOException — Scanner handles any stream errors internally.

Scanner Method Cheat Sheet

  • next() — reads the next single word (stops at a space)
  • nextLine() — reads the entire line, including spaces
  • nextInt() — reads a whole number (like 42)
  • nextDouble() — reads a decimal number (like 3.14)
  • hasNextInt() — checks if the next thing is a whole number (returns true or false, does NOT consume it)

Watch Out: Mixing nextInt() and nextLine()

When you call nextInt(), it reads the number but leaves an invisible "Enter" key sitting in the buffer. If you then call nextLine(), it reads that leftover "Enter" and gives you an empty string instead of the user's input.

Fix: call scnr.nextLine() once right after nextInt() to throw away that leftover Enter.

How to fix the nextInt + nextLine problem
int age = scnr.nextInt();
scnr.nextLine();                    // throw away the leftover Enter
String fullName = scnr.nextLine();  // now this works correctly
