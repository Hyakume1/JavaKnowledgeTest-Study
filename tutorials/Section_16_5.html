Why Make Your Own Exception Types?

You've been using catch (Exception e) to catch errors. But catching Exception catches everything — even errors you didn't expect or intend to handle. That can hide bugs.

A better approach is to create your own specific exception types so that each catch block handles exactly one clearly-named problem. This makes your code easier to read and your error messages much more helpful.

How to Create a Custom Exception

You create a custom exception by making a new class that extends Exception. Inside the constructor, call super(message) to set the error message. That's it — it's just a class that inherits everything an Exception can do.

Two custom exception classes
public class InvalidNegativeInputException extends Exception {
    public InvalidNegativeInputException(String varName) {
        super("Variable " + varName + " is negative.");
    }
}

public class NaNException extends Exception {
    public NaNException(String varName) {
        super("Variable " + varName + " is NaN (Not a Number).");
    }
}

Using Custom Exceptions

When a method might throw your custom exception, you must say so in the method signature using the throws keyword. This is Java's way of warning callers: "hey, this method might throw this error — be ready for it."

DensityCalculator.java — a separate handler for each error type
import java.util.Scanner;

public class DensityCalculator {

    public static double getPositiveValue(Scanner scnr, String name)
            throws InvalidNegativeInputException {
        System.out.print("Enter " + name + ": ");
        double val = scnr.nextDouble();
        if (val < 0.0) {
            throw new InvalidNegativeInputException(name);
        }
        return val;
    }

    public static double getDensity(Scanner scnr)
            throws InvalidNegativeInputException, NaNException {
        double mass   = getPositiveValue(scnr, "mass");
        double volume = getPositiveValue(scnr, "volume");
        double density = mass / volume;
        if (Double.isNaN(density)) {
            throw new NaNException("density");
        }
        return density;
    }

    public static void main(String[] args) {
        Scanner scnr = new Scanner(System.in);
        try {
            System.out.printf("Density: %.4f%n", getDensity(scnr));
        } catch (InvalidNegativeInputException e) {
            System.out.println("Input error: " + e.getMessage());
        } catch (NaNException e) {
            System.out.println("Math error: " + e.getMessage());
        }
    }
}

Checked vs Unchecked Exceptions

There are two kinds of exceptions in Java:

  • Checked exceptions — extend Exception. Java forces you to either handle them with try-catch OR declare them with throws in the method signature. Examples: IOException, FileNotFoundException, and any custom class you make by extending Exception.
  • Unchecked exceptions — extend RuntimeException. Java does NOT force you to handle or declare these. They usually mean a programming bug. Examples: NullPointerException, ArrayIndexOutOfBoundsException, InputMismatchException.

Custom exceptions usually extend Exception (checked) when the error is something the caller should be expected to handle, or extend RuntimeException (unchecked) for programming mistakes that should never happen in a correctly-written program.
