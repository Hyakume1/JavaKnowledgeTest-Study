The Too-Broad Catch Problem

You've seen that catch (Exception e) catches everything. That sounds convenient — but it's actually a trap! When your catch block is too broad, it will swallow exceptions you didn't intend to handle, hiding bugs that should crash your program loudly.

Here's the scenario: a program has a getDensity() method that intentionally throws two custom exceptions:

  • InvalidNegativeInputException — if mass or volume is negative
  • NaNException — if the computed density is NaN (0.0 / 0.0)
Too-broad catch — catches unintended exceptions
try {
    System.out.printf("Density: %.4f%n", getDensity(scnr));
} catch (Exception e) {
    // BAD: catches EVERYTHING — including exceptions you didn't plan for
    System.out.println("Error: " + e.getMessage());
}

Unintended Catches — Concrete Examples

Let's trace what happens with different inputs:

Input "20 twenty": nextDouble() reads 20.0 fine. Then it tries to read "twenty" as a number — that's not a number! This throws an InputMismatchException. But the code only has catch (Exception e)... which catches it. The user sees "Error: null" (InputMismatchException has a null message). This is wrong — InputMismatchException is a bug in the input handling, not an intended exception path!

Input: scnr is null: If scnr (the Scanner) was never initialized and is null, then calling scnr.nextDouble() throws a NullPointerException. Again, the too-broad catch (Exception e) grabs it. The program "handles" this silently instead of alerting you that you have a programming bug.

The Wrong Order — Unreachable Catch Blocks

Because Exception is the parent of all exceptions, putting it first means every exception matches it. Any catch blocks below it become dead code that can never run.

Wrong order — Exception first, everything else unreachable
// ✗ WRONG — catch(Exception) first swallows everything
catch ( Exception e )              { }   // catches ALL exceptions here
catch ( InputMismatchException e ) { }   // never reached — dead code!
catch ( NullPointerException e )   { }   // never reached — dead code!

Does It Handle the NaN Exception Separately?

Sometimes a catch arrangement looks like it handles things separately but doesn't. Let's check two examples:

Does this handle NaNException separately?
// Arrangement 1:
catch ( InputMismatchException e ) { }
catch ( Exception e ) { }
// → NO. InputMismatchException has its own catch, but NaNException falls into
//   catch(Exception) alongside NullPointerException and everything else.
//   NaN is NOT handled separately from all other exceptions.

// Arrangement 2:
catch ( InvalidNegativeInputException e ) { }
catch ( NaNException e ) { }
catch ( Exception e ) { }
// → YES. Each intended exception has its own specific catch block.
//   catch(Exception) only handles truly unexpected errors.

The Right Way — Specific Catches First

The golden rule: most specific exceptions first, most general last.

Correct arrangement — specific before general
try {
    System.out.printf("Density: %.4f%n", getDensity(scnr));
} catch (InvalidNegativeInputException e) {
    System.out.println("Input error: " + e.getMessage());
} catch (NaNException e) {
    System.out.println("Math error: " + e.getMessage());
} catch (Exception e) {
    // Only reaches here for truly unexpected exceptions
    System.out.println("Unexpected error: " + e.getMessage());
}

With this arrangement:

  • Negative input → caught by InvalidNegativeInputException handler only
  • NaN result → caught by NaNException handler only
  • NullPointerException (programming bug) → falls through to catch(Exception) where you know it's unexpected
  • InputMismatchException (wrong input type) → falls through to catch(Exception) where you know it's unexpected

Summary

  • Using catch(Exception) alone catches EVERYTHING — including bugs you didn't intend to handle
  • Specific catches MUST come before broad catches — otherwise the specific ones are unreachable
  • A null Scanner causes NullPointerException — a bug that broad Exception handling hides
  • Wrong input type (like a word when a number is expected) causes InputMismatchException — also hidden by broad Exception handling
