When a File Doesn't Exist — FileNotFoundException

When you try to open a file for reading or writing, that file might not actually be there. If that happens, Java throws a FileNotFoundException right at the line where you try to open it.

Important: FileNotFoundException is only about finding or opening the file. If the file opens fine but contains unexpected data (like letters where you expected numbers), that's an InputMismatchException — a completely different exception. FileNotFoundException does NOT cover bad data inside a valid file.

File Path Rules

When your file is in the same directory as your program, just use the filename with no path prefix:

File in the same directory — use just the filename
fileInStream = new FileInputStream("file1.txt");     // ✓ correct (same directory)
fileInStream = new FileInputStream("my-files/file1.txt"); // ✗ wrong path prefix
fileInStream = new FileInputStream("FILES/file1.txt");    // ✗ wrong path prefix

FileInputStream Must Be Inside the try Block

The FileInputStream constructor is the line that can throw FileNotFoundException. It must be inside the try block to be caught. If you put it before the try, the exception won't be caught.

Correct placement
// ✓ CORRECT — FileInputStream is inside the try
try {
    fileScnr = new Scanner(new FileInputStream(fileName));
} catch (FileNotFoundException e) {
    System.out.println("File not found.");
}

// ✗ WRONG — FileInputStream is outside the try, exception won't be caught
fileStream = new FileInputStream(fileName);  // throws here — not caught!
try {
    fileScnr = new Scanner(fileStream);
} catch (FileNotFoundException e) { }

Resource Leaks — Which Files Are Left Open?

This is one of the trickiest parts of file exception handling. When an exception is thrown, execution jumps to the catch block immediately. Any files that were opened before the exception was thrown stay open — they are NOT automatically closed. This is called a resource leak.

Consider this code that opens two files without try-with-resources:

Code WITHOUT try-with-resources
Scanner   fileScanner = null;
PrintWriter fileWriter  = null;

try {
    fileScanner = new Scanner(new FileInputStream("input.txt"));   // Step 1
    fileWriter  = new PrintWriter(new FileOutputStream("output.txt")); // Step 2
    int x = fileScanner.nextInt();                                  // Step 3
    fileWriter.println(x * 2);                                      // Step 4
} catch (FileNotFoundException e) {
    System.out.println("Could not open a file.");
}
// No 'finally' block — no guaranteed cleanup!

Let's trace three different failure scenarios:

  • Scenario A: input.txt not found → Step 1 throws FileNotFoundException immediately. Neither file ever opened. The catch block runs. Zero files left open.
  • Scenario B: input.txt opens fine, output.txt can't open → Step 1 succeeds (input.txt is open). Step 2 throws FileNotFoundException. We jump to catch. fileScanner never got closed! input.txt is left open — resource leak!
  • Scenario C: both files open, then InputMismatchException while reading → Steps 1 and 2 both succeed. Step 3 throws InputMismatchException. But the catch block only catches FileNotFoundException! The InputMismatchException is not caught — it propagates up, and neither file is closed. Both files are left open!

The finally Block — Guaranteed Cleanup

A finally block runs no matter what — even if an exception was thrown, even if the try block ran perfectly. Put file-closing code here.

FileDataAverage.java — finally ensures cleanup
Scanner   fileScanner = null;
PrintWriter fileWriter  = null;

try {
    fileScanner = new Scanner(new FileInputStream("input.txt"));
    fileWriter  = new PrintWriter(new FileOutputStream("output.txt"));
    // ... read and write data ...
} catch (FileNotFoundException e) {
    System.out.println("Could not open a file: " + e.getMessage());
} finally {
    if (fileScanner != null) fileScanner.close();  // always runs
    if (fileWriter  != null) fileWriter.close();   // always runs
}

try-with-resources — The Modern Easy Way

Java has a cleaner way to handle file closing automatically: declare the file inside the parentheses of try(...). Java then closes it automatically when the block ends — no finally needed.

try-with-resources syntax
// ✓ CORRECT — FileInputStream declared in try-with-resources
try (FileInputStream inStrm = new FileInputStream(fName);
     Scanner scnr = new Scanner(inStrm)) {
    // ... use scnr ...
} catch (FileNotFoundException e) {
    // both inStrm and scnr are auto-closed when this block ends
}

// ✗ WRONG — forward reference: using inStrm before it's declared!
try (Scanner scnr = new Scanner(inStrm);          // inStrm doesn't exist yet!
     FileInputStream inStrm = new FileInputStream(fName)) {
}

Order matters in try-with-resources! You must declare resources in the order they depend on each other. inStrm must be declared before scnr because scnr needs inStrm. If you put scnr first and reference inStrm inside it, Java gives a compile error — it's a forward reference.
