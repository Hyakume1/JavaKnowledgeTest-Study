What Is an Exception?

An exception is an error that happens while your program is running — something unexpected that Java doesn't know how to continue from. If an exception is not handled, the program stops immediately and prints an ugly error message.

InputMismatchException — The Most Common Beginner Exception

When you call scnr.nextInt(), Java expects the next token to be a whole number. If it's anything else — a word, a decimal, a negative decimal — Java throws an InputMismatchException.

Let's trace through what happens with different inputs for this code:

Reading two integers
int num1 = scnr.nextInt();
int num2 = scnr.nextInt();
  • Input: 5 10 → OK. 5 is a valid int, 10 is a valid int. No exception.
  • Input: 43 -7 → OK. Negative integers are valid too. No exception.
  • Input: 12 34 → OK. Extra spaces don't matter. No exception.
  • Input: 12 34 two → OK. nextInt() reads 12, then 34. "two" is never read. No exception.
  • Input: 17 two 3 → CRASH! nextInt() reads 17 fine, then tries to read "two" as an integer. "two" is a word, not a number → InputMismatchException!
  • Input: 35 45.1 → CRASH! nextInt() reads 35 fine, then tries to read "45.1" as an integer. Integers can't have decimal points → InputMismatchException!

Key rule for nextInt(): Any token that is not a plain whole number (no decimals, no letters) will cause an InputMismatchException.

InputMismatchException with nextDouble()

nextDouble() reads decimal numbers. It accepts things like 3.14, -5.5, and +5 — but NOT numbers with commas in them (even if humans write them that way).

Reading a double
double distMiles = scnr.nextDouble();
  • Input: 3.14 → OK. Valid decimal number.
  • Input: -5.5 10 → OK. Reads -5.5.
  • Input: +5 0 → OK. Reads +5.
  • Input: 1,0 23 → CRASH! Java sees "1,0" as one token. The comma inside is not valid for Java's double format → InputMismatchException! (Even though many countries write "1,0" to mean 1.0)
  • Input: 1,000 23000 → CRASH! Same reason — comma inside the number is not valid.

Catching Exceptions with try / catch

Instead of letting your program crash, you can catch the exception and handle it gracefully:

Basic try-catch structure
try {
    // code that might cause an error
} catch (ExceptionType varName) {
    // code that runs if the error happens
}
LightTravelTime.java — keeps asking until valid input
import java.util.Scanner;
import java.util.InputMismatchException;

public class LightTravelTime {
    public static void main(String[] args) {
        Scanner scnr = new Scanner(System.in);
        double distMiles = 0.0;
        boolean needInput = true;

        while (needInput) {
            System.out.print("Enter a distance in miles: ");
            try {
                distMiles = scnr.nextDouble();
                needInput = false;           // it worked! stop the loop
            } catch (InputMismatchException e) {
                scnr.nextLine();             // clear out the bad input
                System.out.println("Please enter a number.");
            }
        }

        double travelTime = distMiles / 186282.0;
        System.out.printf("Light travels %.1f miles in %.6f seconds.%n",
                          distMiles, travelTime);
    }
}
Sample run 
Enter a distance in miles: hello
Please enter a number.
Enter a distance in miles: 1000000
Light travels 1000000.0 miles in 5.368205 seconds.

Common Exception Types

  • InputMismatchException — Scanner got the wrong type (text instead of a number, decimal instead of int, comma in double)
  • ArithmeticException — illegal math (e.g., dividing by zero)
  • NullPointerException — tried to use a variable that is null
  • ArrayIndexOutOfBoundsException — tried to access an array position that doesn't exist
  • NumberFormatException — tried to convert text like "abc" into a number
  • FileNotFoundException — tried to open a file that doesn't exist
