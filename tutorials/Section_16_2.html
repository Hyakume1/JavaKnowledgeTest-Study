Throwing Your Own Exceptions

So far exceptions happened automatically when something went wrong. But you can also manually create and throw an exception yourself using the throw keyword. This is useful for input validation — checking if the user's input makes sense before you use it.

The correct syntax is: throw new Exception("message here");

  • You need throw — this triggers the exception
  • You need new — this creates the Exception object
  • The message goes inside the parentheses as a String
Correct throw syntax
throw new Exception("Invalid ID");           // ✓ correct
// throw Exception("Invalid ID");            // ✗ wrong — missing 'new'
// Exception.throw("Invalid ID");            // ✗ wrong — throw is a keyword, not a method
// new Exception("Invalid ID");              // ✗ wrong — creates exception but doesn't throw it

getMessage() vs toString()

Every exception object has two ways to get its description:

  • getMessage() — returns only the message you passed in. Short and clean.
  • toString() — returns the exception class name plus the message. Much longer.
getMessage() vs toString()
try {
    throw new Exception("Invalid ID");
} catch (Exception excpt) {
    System.out.println(excpt.getMessage());  // → "Invalid ID"
    System.out.println(excpt.toString());    // → "java.lang.Exception: Invalid ID"
    System.out.println(excpt);               // → same as toString()
}

If you want to print just the message, use getMessage(). If you accidentally use toString() or just excpt, you get the class name prepended — more info than you asked for.

Which catch Block Matches Which throw?

Java matches the throw to the exact type thrown (or a parent type). When you write throw new Exception(...), you're throwing a plain Exception object — NOT an InputMismatchException or ArithmeticException. Only a catch (Exception ...) block can catch it.

Only catch(Exception) matches throw new Exception(...)
throw new Exception("0.0 / 0.0 is not a number");

// Which catch matches?
catch (Exception excpt)            { }  // ✓ YES — matches base Exception
catch (InputMismatchException e)   { }  // ✗ NO — wrong type
catch (ArithmeticException e)      { }  // ✗ NO — wrong type
catch (NullPointerException e)     { }  // ✗ NO — wrong type

The CircleRadiusCalculator Trace

Let's trace through what happens with different inputs:

CircleRadiusCalculator.java
import java.util.Scanner;
import java.util.InputMismatchException;

public class CircleRadiusCalculator {
    public static void main(String[] args) {
        Scanner scnr = new Scanner(System.in);
        try {
            System.out.print("Enter radius: ");
            double radius = scnr.nextDouble();
            if (radius <= 0.0)
                throw new Exception("Invalid area.");
            double area = Math.PI * radius * radius;
            System.out.printf("Area: %.2f%n", area);
        } catch (InputMismatchException e) {
            System.out.println("Invalid input.");
        } catch (Exception e) {
            System.out.println(e.getMessage());
        }
    }
}

Trace for input "pi": nextDouble() tries to read "pi" as a number — but "pi" is a word, not a number. So it throws an InputMismatchException. Java checks the catch blocks in order: the first one is catch (InputMismatchException e) — that matches! So it prints "Invalid input." The second catch block never runs.

Trace for input "-1.0": nextDouble() reads -1.0 fine. Then radius <= 0.0 is true, so we hit throw new Exception("Invalid area."). This throws a plain Exception. The first catch (InputMismatchException) doesn't match. The second catch (Exception) does — so it prints "Invalid area."

Order of catch Blocks Matters!

Always put more specific exception types first. Here's why this is critical:

InputMismatchException is a subclass of Exception. If you put catch (Exception e) first, it catches EVERYTHING — including InputMismatchException. The specific catch block below it would become dead code that can never run (and Java will give you a compile error).

Correct order: specific before general
// ✓ CORRECT — specific first, general last
catch (InputMismatchException e) { System.out.println("Invalid input."); }
catch (Exception e)              { System.out.println(e.getMessage()); }

// ✗ WRONG — Exception catches InputMismatchException too; second block unreachable
catch (Exception e)              { System.out.println(e.getMessage()); }
catch (InputMismatchException e) { System.out.println("Invalid input."); }

One catch (Exception e) block by itself does catch both Exception and InputMismatchException — because InputMismatchException IS-A Exception. But then you handle them both the same way.

Using throw for Input Validation

DensityCalculator.java — throwing for negative values
import java.util.Scanner;

public class DensityCalculator {
    public static void main(String[] args) {
        Scanner scnr = new Scanner(System.in);

        try {
            System.out.print("Enter mass (kg): ");
            double mass = scnr.nextDouble();
            if (mass < 0.0) {
                throw new Exception("Invalid mass: cannot be negative.");
            }

            System.out.print("Enter volume (m^3): ");
            double volume = scnr.nextDouble();
            if (volume < 0.0) {
                throw new Exception("Invalid volume: cannot be negative.");
            }

            System.out.printf("Density: %.4f kg/m^3%n", mass / volume);

        } catch (Exception e) {
            System.out.println("Error: " + e.getMessage());
        }
    }
}
