How to Write a Recursive Method

There's a reliable two-step approach to writing any recursive method:

  1. Write the base case first. Ask yourself: what is the simplest possible input that I can answer immediately, without needing to call myself again? There may be more than one base case.
  2. Write the recursive case. Assume the method magically works correctly for smaller inputs, and use it to solve the current input. Make sure every recursive call moves toward a base case — otherwise you'll recurse forever.

Example: Factorial

Factorial of N (written N!) means multiplying N × (N-1) × ... × 2 × 1. The base case is: 0! = 1 (just a fact we know by definition).

Factorial.java
public class Factorial {
    // This private helper does the actual recursion
    private static int calcFactorialHelper(int n) {
        if (n <= 1) {              // base case: 0! = 1! = 1
            return 1;
        } else {                   // recursive case: n! = n × (n-1)!
            return n * calcFactorialHelper(n - 1);
        }
    }

    // This public method is what callers use — it validates input first
    public static int calcFactorial(int n) {
        if (n < 0) {
            throw new IllegalArgumentException("Factorial undefined for " + n);
        }
        return calcFactorialHelper(n);
    }

    public static void main(String[] args) {
        for (int i = 0; i <= 7; i++) {
            System.out.printf("%d! = %d%n", i, calcFactorial(i));
        }
    }
}
Output 
0! = 1
1! = 1
2! = 2
3! = 6
4! = 24
5! = 120
6! = 720
7! = 5040

Outer Method / Inner Helper Pattern

It's very common to split recursion across two methods:

  • Outer method (public) — the one callers use. It checks for bad input and sets up any initial values before calling the helper.
  • Inner helper method (private) — does the actual recursion. It may need extra parameters (like a depth counter) that callers shouldn't have to worry about.

Should I Use Recursion or a Loop?

Ask yourself these two questions:

  1. Does this problem naturally split into smaller versions of itself? (Trees, searching, generating all combinations → usually yes. Simple counting → usually no.)
  2. Is the recursive version easier to read and understand than a loop?

Factorial can be done with a loop just as easily. Binary search, generating all permutations, and traversing trees are cases where recursion is really the cleaner choice.

Common Errors

  • Missing base case — infinite recursion → StackOverflowError.
  • Base case never reached — the recursive call passes the same value instead of a smaller one (e.g., calling helper(n) instead of helper(n-1)).
  • Forgetting to return the recursive result — calling helper(n-1) without the return keyword silently throws away the answer.
