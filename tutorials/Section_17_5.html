Writing a Recursive Method — Two Steps

A reliable approach to writing recursive methods:

  1. Identify and write the base case first. What is the simplest possible input that can be answered immediately, without a recursive call? There may be more than one base case.
  2. Write the recursive case. Assume the method works correctly for smaller inputs and use it to solve the current input. Make sure every recursive call moves toward a base case.

Example: Factorial

N! (N factorial) = N × (N−1) × ... × 2 × 1. Base case: 0! = 1.

Factorial.java
public class Factorial {
    // Inner (helper) method — the actual recursion
    private static int calcFactorialHelper(int n) {
        if (n <= 1) {              // base case: 0! = 1! = 1
            return 1;
        } else {                   // recursive case: n! = n × (n-1)!
            return n * calcFactorialHelper(n - 1);
        }
    }

    // Outer (public) method — validates input before recursing
    public static int calcFactorial(int n) {
        if (n < 0) {
            throw new IllegalArgumentException("Factorial undefined for " + n);
        }
        return calcFactorialHelper(n);
    }

    public static void main(String[] args) {
        for (int i = 0; i <= 7; i++) {
            System.out.printf("%d! = %d%n", i, calcFactorial(i));
        }
    }
}
Output 
0! = 1
1! = 1
2! = 2
3! = 6
4! = 24
5! = 120
6! = 720
7! = 5040

Outer / Inner Method Pattern

It is common to split recursion across two methods:

  • Outer method — the public-facing entry point. Validates input and initializes any extra parameters needed by the recursion.
  • Inner (helper) method — does the actual recursion. It may have extra parameters (like an index, an accumulator, or a depth counter) that are only meaningful during the recursion and should not be exposed to callers.

Should I Use Recursion or a Loop?

Ask yourself two questions before choosing recursion:

  1. Does the problem naturally decompose into smaller versions of itself? (Trees, search, combinatorics → yes. Simple counting → no.)
  2. Is the recursive solution clearer or easier to write than the loop version?

Factorial has a natural recursive definition, but a loop is just as clear and uses less memory. Binary search, permutations, and tree traversal are examples where recursion genuinely shines.

Common Errors

  • Missing base case — infinite recursion → StackOverflowError.
  • Base case never reached — the recursive call doesn't reduce the problem (e.g., calling helper(n) instead of helper(n-1)).
  • Not returning the recursive result — calling helper(n-1) without return silently discards the answer.
