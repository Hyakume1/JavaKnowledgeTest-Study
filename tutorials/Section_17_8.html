How Java Keeps Track of Recursive Calls

Every time a method is called, Java sets aside a small block of memory to hold that method's variables and keep track of where to return when it's done. This block is called a stack frame.

Think of it like a stack of sticky notes. When you call a method, you write a new sticky note and put it on top of the pile. When the method returns, you peel it off. With recursion, each unfinished call keeps its sticky note on the pile — so if you go 10 levels deep, there are 10 sticky notes stacked up at once.

Stack Overflow: When the Stack Gets Too Full

The pile of sticky notes has a limit. If your recursion goes too deep — thousands or millions of calls without returning — you run out of space. Java throws a StackOverflowError and crashes the program.

Triggering a StackOverflowError
// This will crash — no base case, calls itself forever
public static int bad(int n) {
    return n + bad(n - 1);  // n never reaches a stopping point
}

// This will also crash for very large n — too many frames stacking up
public static long factorial(long n) {
    if (n <= 1) return 1;
    return n * factorial(n - 1);  // fine for n=10000, crashes for n=100000+
}

What Makes a Stack Frame Bigger?

The more local variables a method declares — especially large arrays or strings — the more memory each stack frame takes. Methods with fewer variables can recurse much deeper before hitting the limit.

How to Avoid Stack Overflow

  • Add a base case — the most common fix. Make sure every path through the code eventually reaches it.
  • Convert to a loop — loops don't add stack frames, so they can repeat millions of times without any risk. Use this when inputs can be very large.
  • Use an explicit stack data structure — replace the call stack with a Stack<T> object in memory (the heap), which can hold far more data than the call stack.
  • Increase the JVM stack size — you can pass -Xss4m when running Java to give the call stack more memory. This is a workaround, not a real fix.
Safe loop version of factorial — no stack overflow risk
public static long factorialIterative(int n) {
    long result = 1;
    for (int i = 2; i <= n; i++) {
        result *= i;
    }
    return result;
}

Rule of thumb: If the maximum recursion depth is small and known (under a few thousand), recursion is perfectly fine. If depth depends on user input or file size that could be huge, use a loop instead.
