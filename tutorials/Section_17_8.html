The Call Stack

Every method call reserves a block of memory called a stack frame on the program's call stack. The frame stores the method's local variables, parameters, and the return address. When the method returns, its frame is removed.

With recursion, each unresolved call keeps its frame on the stack until the base case is reached and the chain of returns begins. Deep recursion therefore keeps many frames alive simultaneously.

Stack Overflow

If recursion goes too deep, the stack runs out of space. Java throws a StackOverflowError and the program crashes.

Triggering a StackOverflowError
// This will crash — infinite recursion (no base case)
public static int bad(int n) {
    return n + bad(n - 1);  // n never reaches a base case
}

// This will also crash for large n — too many frames
public static long factorial(long n) {
    if (n <= 1) return 1;
    return n * factorial(n - 1);  // fine for n=10000, crashes for n=100000+
}

What Makes Frames Large?

The size of a stack frame depends on the number and size of local variables. A method with many large arrays or strings declared locally will consume stack memory much faster and may overflow at a shallower depth.

How to Avoid Stack Overflow

  • Add a base case — the most common fix. Ensure every possible input eventually reaches it.
  • Convert to iteration — a loop uses constant stack space regardless of input size. Use this when the input can be very large.
  • Use an explicit stack — replace the call stack with a Stack<T> data structure on the heap, which can grow much larger than the call stack.
  • Increase the JVM stack size — pass -Xss4m to the JVM to increase the thread stack size (a temporary workaround, not a fix).
Iterative alternative to recursive factorial
// No risk of stack overflow regardless of input size
public static long factorialIterative(int n) {
    long result = 1;
    for (int i = 2; i <= n; i++) {
        result *= i;
    }
    return result;
}

As a rule of thumb: if the maximum recursion depth is bounded and small (under a few thousand), recursion is fine. If it depends on unbounded user input or file size, prefer iteration.
